# Filosofía

Cuatro principios sostienen todo el sistema. Si un día dudas de un paso, vuelve aquí: el paso existe para servir a uno de estos cuatro.

Identidad intocable

Los Invariantes dicen qué ES el proyecto (2–4 máximo). Prueba de fuego: si nombra una herramienta, librería o topología, no es un invariante — es una decisión de arquitectura, y esas se cambian con una línea en el log.

→ lección: el blueprint que fija el "cómo" se vuelve camisa de fuerza.

Goal medible

El Goal es un número que el producto puede fallar ("el agente recrea ≥75% de la spec"), nunca una descripción ("construir un sistema que..."). Es el juez de cada fase y de cada ciclo.

→ lección: sin norte medible, la ejecución produce vacío conformante.

Piso por plan, vida por ciclos

La infraestructura (el piso) es enumerable: se planifica y se ejecuta por fases. La vida real del proyecto son ciclos: duda → objetivo → medición → cierre.

→ lección: todo proyecto real terminó viviendo en ciclos; el sistema ahora los soporta.

Medición o teatro

Nada se declara terminado sin verificarse contra disco y medirse contra el Goal. Un informe que renderiza pero no dice nada, una métrica inflada por desbalance: eso es report theater, y el sistema lo caza explícitamente.

→ lección: COMPLETED de memoria = fases falsas; métrica sin balancear = 0.84 mentiroso.
//02 · el orden de los pasos

# El flujo, en orden

De cero a proyecto vivo. Las etiquetas [TÚ] marcan lo que tecleas; [AUTO] lo que el sistema dispara solo.

00

Instalar la base TÚ

Clona/instala el plugin en el proyecto nuevo e inicializa. /setup hace solo 2 preguntas (nombre y stack), crea las 5 carpetas mínimas, copia rules/skills/ agents/hooks e instala promptloop.sh.

terminal
# dentro de Claude Code, en el repo nuevo:
/plugin marketplace add OilCoder/claude-project-base
/plugin install claude-project-base
/setup

Recomendación: no crees src/, tests/, data/ a mano — nacen cuando el proyecto los pide.

01

El ancla: /blueprint TÚ

Se hace una sola vez. Te pregunta el tipo de proyecto (ml-research / data-pipeline / other) y la ambición:

decisión de ambición
ACOTADO  hipótesis + dataset + protocolo conocibles hoy
         → charter-lite: 2 documentos, 1 sesión de entrevista
ABIERTO  el "cómo" es la incógnita a investigar
         → ciclo completo de 5 pasos (Charter → Contexto → Diseño → Plan → Validación)

El resultado es el Charter con Goal medible, Invariantes de identidad, Architecture decisions (revisables) y Pilares — y siembra planning/PLAN.md.

Recomendación: aprueba cada documento con escrutinio real. Aprobar 9 docs en un día convierte el ancla en lastre.

02

El piso: promptloop.sh AUTO

Las fases enumerables del plan (esqueleto E2E, verificación, infraestructura) las ejecuta el loop autónomo — contexto fresco por fase, un commit por fase, y se niega a arrancar si el ancla no existe.

tu shell (no dentro de claude)
git switch -c loop/floor
bash .claude/scripts/promptloop.sh 3          # empieza pequeño
# frenos opcionales:
MAX_TURNS=80 MAX_BUDGET_USD=5 bash .claude/scripts/promptloop.sh 8

Por qué antes de /goal: la libertad solo funciona sobre un suelo verificado. Goal-run sin piso = basura con confianza.

03

Checkpoint de contacto AUTO

La primera fase que toca datos reales trae sembrada la tarea: "¿alguna Architecture decision murió al tocar los datos?". Si algo murió: una línea en el log de decisiones y sigues. Si murió la identidad: /blueprint --v2.

Recomendación: espera que algo pivote aquí — la mitad del diseño solo es descubrible construyendo. No es fracaso, es el sistema funcionando.

04

La vida: ciclos con /goal TÚ

Piso completo → cada duda o auditoría abre un ciclo en planning/cycles/NN_<slug>.md. El archivo trae el comando listo:

goal-run
/goal <objetivo del ciclo> respetando los Pillars de planning/PLAN.md;
      para cuando <la medición de cierre se cumpla o esté probadamente bloqueada>

El ciclo se cierra midiendo contra el Goal — números, no adjetivos. Sin medición no hay cierre.

05

/checkpoint — tu único ritual TÚ

En cada hito tecleas /checkpoint y el sistema encadena solo: plan + ciclo abierto + documentación + material de estudio + bitácora + commit (+ push/PR si confirmas). Es lo único que tienes que recordar.

06

Repetir ciclos hasta que el Goal no deje dudas

El proyecto no "termina cuando se acaban las tareas" — termina cuando la medición contra el Goal ya no te deja dudas que valgan otro ciclo. Cada ciclo nuevo nace de la medición del anterior.

El hilo completo: Charter (Goal + Pilares) → PLAN.md (piso) → promptloop.sh (ejecuta con el Goal como juez) → cycles/ (cada uno con su comando /goal derivado del Charter) → /checkpoint (cierra midiendo). El objetivo nunca se re-redacta: se hereda.
//03 · las piezas del charter

# El ancla de dos niveles

El Charter separa lo constitucional de lo revisable. Confundir los niveles es lo que convierte una guía en camisa de fuerza.

SecciónQué esCambiarla cuesta
## GoalResultado medible sobre el artefacto, redactado como condición verificable (apto para /goal)Re-blueprint — es el norte
## InvariantsIdentidad: qué ES el proyecto (2–4 máx). Nunca una herramienta/topologíaDecisión logueada + gate de aprobación
## Non-goalsLo que el proyecto explícitamente NO haráDecisión logueada + aprobación
## PillarsRieles de decisión: qué no decide Claude · qué construye libre · ambiguo → registrar, no adivinarDecisión logueada + aprobación
## Architecture decisionsEl "cómo" actual: frameworks, topologías, métodos, modelosUna línea en el log. Sin ceremonia.
Prueba de fuego: ¿la frase nombra una herramienta, librería o topología? Entonces NO es un Invariante. La convicción con la que lo dices no lo vuelve identidad — la arquitectura casi nunca sobrevive al contacto con los datos reales, y no pasa nada.
//04 · qué loop cuándo

# Los tres loops

Tres herramientas para tres momentos. No compiten — se turnan.

MomentoHerramientaPor qué esa
Piso — fases enumerables, tú ausente promptloop.sh (nuestro) Contexto fresco por fase, gates duros (ancla, rama, BLOCKED, no-progreso), un commit por fase, frenos MAX_TURNS/MAX_BUDGET_USD. Sin equivalente nativo.
Ciclos / goal-runs — objetivo medible, tú cerca /goal (nativo) Corre turno tras turno hasta que la condición verificable se cumple. El contexto acumulado es un activo: va entendiendo el artefacto.
Corridas largas — entrenamientos, batches nocturnos /loop 30m "..." (nativo) Babysitting por intervalo: revisa el run.log, detecta el proceso muerto a las 2 AM en vez de a las 9. Con /bg sobrevive al cierre de la terminal.

//05 · la unidad de vida

# Anatomía de un ciclo

Un ciclo = un objetivo = un archivo. Corto, cerrable, medido.

planning/cycles/02_canales_fisicos.md
# Cycle 02 — canales físicos

## Origin
Auditoría 06-09: errores difusos en yacimiento (ratio ≈ 1) — señal débil,
no no-unicidad. Ver bitácora 2026-06-09.

## Objective
Subir balanced accuracy de yacimiento de 0.61 a ≥0.64 en el test estratificado,
sin regresión en extrapolación.

## Goal-run command
/goal subir balanced acc de yacimiento a ≥0.64 (test estratificado) respetando
      los Pillars de planning/PLAN.md; para cuando la medición lo confirme
      o esté probadamente bloqueada

## Tasks
- [x] Canal sep (Δp−Δp′) con test analítico (2026-06-09)
- [>] Gate dual-seed 5ch vs control 3ch

## Close — measurement against the Goal
(se llena al cerrar: números vs el Objective — sin esto el ciclo NO está cerrado)
Un objetivo por ciclo. Si a mitad de ciclo aparece un segundo objetivo, ese abre el ciclo siguiente — no estira este. Y el plan maestro solo lleva el índice: una línea por ciclo, nunca más fases anexadas.
//06 · cuando algo cambia

# Taxonomía del cambio

Tres tamaños de cambio, tres costos. Saber cuál es cuál evita tanto la parálisis como la deriva.

CambioEjemploQué haces
De forma (arquitectura) Cambiar librería, topología del pipeline, método, modelo Una línea en el log de decisiones. Sigues trabajando.
Nuevo esfuerzo (duda/mejora) La auditoría reveló un hueco; quieres subir una métrica Abres un ciclo nuevo con su objetivo medible.
De fondo (identidad) Los Invariantes ya no describen lo que el proyecto es /blueprint --v2: congela la suite, re-blueprintea con gates completos, re-siembra el plan.
//07 · anti-patrones — cada uno costó caro una vez

# Qué NO hacer

Diez prohibiciones. Ninguna es hipotética: cada una tiene nombre, fecha y cicatriz.

No eleves arquitectura a Invariante

"El orquestador es LangGraph, nunca un LLM" parecía identidad. Duró 2 días. Cuando el "cómo" es constitución, cada aprendizaje legítimo se siente como violación y el blueprint se vuelve camisa de fuerza.

caso real: petro-agent — blueprint v1 muerto en 48h; el único invariante verdadero sobrevivió ambos pivotes.

No corras /goal sin piso

El goal-run funciona porque debajo hay librería verificada, gates y trazabilidad. Dársela a un proyecto sin suelo produce basura con confianza: números inventados sin forma de detectarlos.

caso real: el breakthrough del 30-jun funcionó sobre 2 días de infraestructura previa — no a pesar de ellos.

No marques COMPLETED de memoria

COMPLETED se verifica contra disco: los archivos citados existen y el Done when: se demuestra ejecutándolo. La suite verde cubre lo que existe, no lo que se prometió.

caso real: petro-agent D8 — fases enteras marcadas completas citando archivos que no existían.

No cierres un ciclo sin medición

"Quedó bien" no es un cierre. El cierre son números contra el objetivo. Sin medición, el ciclo sigue abierto — lo contrario es report theater con fecha.

caso real: pinn — el trabajo post-plan quedó huérfano: tareas nunca cerradas ni descartadas.

No anexes fases al plan maestro

El PLAN.md cubre el piso. Después, cada esfuerzo abre su ciclo y el plan solo lleva el índice. Un plan que crece para siempre se convierte en archivo histórico que nadie puede leer.

caso real: Deep-PTA — 7 ciclos anexados como fases: 193 líneas de historia densa.

No apruebes el blueprint en batch

La puerta de aprobación solo ancla si aprobar significa escrutinio. Nueve documentos aprobados en un día es un ancla de papel.

caso real: petro-agent — suite completa aprobada el 06-24; pivote total el 06-26.

No dependas de datos que no tienes en la mano

Todo dato externo se clasifica el día cero: público-sin-fricción / con-login / por-digitalizar. Ninguna fase del piso puede depender de un dato no verificado.

caso real: el dataset Volve (login Equinor) descarriló fases en DOS proyectos distintos.

No dejes corridas nocturnas sin niñera

Un batch de horas se vigila con /loop 30m "revisa run.log...". Y persiste artefactos por unidad de trabajo apenas terminan — lo que vive solo en memoria muere con el proceso.

caso real: batch v7 muerto tras el 4º pozo a medianoche; nadie lo supo hasta las 9 AM.

No presentes una métrica de cabecera sin balancear

El protocolo ML exige de serie: métrica balanceada, chequeo de fuga y control pareado (una corrida λ=0 que debe reproducir el baseline). La métrica inflada te engaña primero a ti.

caso real: Deep-PTA — 0.84 de frontera que era 0.61 balanceado (70% de la clase dominante).

No corras promptloop en main ni a lo grande la primera vez

Siempre en rama desechable (loop/...) — el script se niega a correr en main/master — y empieza con MAX=2-3 para ver los gates en acción antes de soltarle 10 iteraciones.

defensa en profundidad: guard de rama + hooks PreToolUse + un commit por fase = todo reversible.
